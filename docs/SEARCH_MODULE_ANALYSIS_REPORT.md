# Звіт аналізу модуля Пошук (Smart Search)

## 1. Що надає пошук користувачу

| Блок | Опис | Корисність |
|------|------|------------|
| **1. Пряма відповідь** | 2–3 речення в body, перше = пряма відповідь | ⭐⭐⭐ Ключовий блок |
| **2. Ключові висновки** | 6–12 impactAreas | ⭐⭐⭐ Скорочена вижимка |
| **3. Регуляторний бекграунд** | Статті, annexes, MDCG, дати чинності | ⭐⭐⭐ База для перевірки |
| **4. Вплив на ситуацію** | Персоналізація під роль/нішу | ⭐⭐⭐ Практична цінність |
| **5. Дії, що потрібні** | Чекліст з пріоритетами [Критичне/Високе/Середнє] | ⭐⭐⭐ Найцінніший для RA |
| **6. Ризики та наслідки** | CE, NB, recall, nonconformity | ⭐⭐⭐ Критично для рішень |
| **7. Рекомендації експерта** | Best practices | ⭐⭐ Допоміжний |
| **8. Обговорення в соцмережах** | Коментарі практиків | ⭐⭐ Сигнали, не джерела |
| **9. Джерела** | 12–20 посилань (EUR-Lex, EC, MDCG) | ⭐⭐⭐ Якщо URL реальні |
| **10. Наступні кроки** | 3 follow-up запитання | ⭐⭐ Навігація |

---

## 2. Релевантність і корисність для юзера

### Що працює добре
- **Структура answer-first** — перше речення має давати пряму відповідь.
- **Персоналізація** — профіль (роль, ніша, device type) передається в промпти.
- **Режими** — deadline, action, document, risk, compliance, business підлаштовують глибину відповіді.
- **Чекліст** — формат [Критичне до 30 днів] / [Високе до 90 днів] / [Середнє] зручний для планування.

### Ризики для релевантності
1. **Немає реального retrieval** — модель генерація, а не пошук. Відповідь базується на знаннях LLM, не на актуальних документах.
2. **Посилання можуть бути неактуальні** — Grok може повертати приблизні URL; LinkValidator перевіряє ~8 links і може їх відкидати.
3. **Social insights** — генеруються LLM, це не реальні коментарі з LinkedIn/X.
4. **Fallback** — при збої API повертається generic seed-картка; вона не прив’язана до конкретного запиту.

## 3. Точність

| Аспект | Оцінка | Коментар |
|--------|--------|----------|
| Відповідь на запит | ⚠️ Залежить від LLM | Модель може “галюцинувати” дати, номери статей |
| Статті / annexes | ⚠️ Середня | Краще для загальних тем (MDR, IVDR), гірше для дуже конкретних MDCG |
| Дедлайни | ⚠️ Середня | Дати можуть застарівати, потрібна перевірка по офіційних джерелах |
| Джерела (URL) | ⚠️ Низька–середня | Часто загальні сторінки; LinkValidator відсікає недоступні |

**Висновок:** Для інформаційної підтримки та орієнтування — корисний. Для юридично обов’язкових рішень — потрібна обов’язкова перевірка по офіційних джерелах.

---

## 4. Чи це корисно юзеру?

**Так, якщо:**
- Потрібно швидко отримати орієнтовну відповідь і план дій.
- Користувач розуміє, що це допоміжний інструмент, а не офіційна юридична порада.
- Використовується як старт для подальшого глибокого дослідження.

**Ні/обережно, якщо:**
- Потрібні гарантовано актуальні дати й номери статей.
- Потрібні реальні посилання на конкретні документи.
- Використовується для compliance-рішень без перевірки.

## 5. Як прогнати 500 запитів (практично)

### Варіант A: Ручна вибірка
- Візьми 50–100 репрезентативних запитів з `search_queries_test_set.json`.
- Прогони по 5–10 на день, оцінюючи кожну відповідь за rubric нижче.

### Варіант B: Скрипт (потребує API key)
- Створити інструмент, який читає `search_queries_test_set.json` і викликає `SearchRepository.search()`.
- Зберігати результати (title, body, links) у JSON/CSV.
- Потім вручну або напівавтоматично оцінити вибірку (наприклад, 100 із 500).

### Варіант C: A/B на реальних юзерах
- Збирати фідбек (корисно/некорисно) після кожного пошуку.
- Аналізувати, які типи запитів дають найкращу і найгіршу оцінку.

## 6. Критерії оцінки відповіді (rubric)

Оцінюй кожну відповідь по шкалі 1–5:

| Критерій | 1 | 3 | 5 |
|----------|---|---|---|
| **Відповідає запиту** | Ні | Частково | Так, прямо |
| **Перше речення — пряма відповідь** | Ні | Не завжди | Завжди |
| **Чекліст дій конкретний** | Загальний | Середній | Конкретні кроки |
| **Ризики релевантні** | Ні | Частково | Так |
| **Джерела корисні** | Ні/порожньо | 1–2 вдалі | 3+ вдалі |
| **Персоналізація** | Відсутня | Базова | Під роль/нішу |

**Поріг прийнятності:** середня оцінка ≥ 3.5 по всіх критеріях.

## 7. Що покращити (пріоритети)

### Високий пріоритет
1. **Grounding** — підключати retrieval (наприклад, RAG з офіційними PDF/HTML) перед генерацією.
2. **Фактчекінг дат** — перевіряти дедлайни по EUR-Lex або EC перед показом.
3. **Позначати fallback** — якщо показано seed-картку, показувати мітку “Офлайн-чернетка” / “Тимчасовий результат”.

### Середній пріоритет
4. **Розширити focus signals** — додати більше термінів у `extractSearchFocusSignals` для кращої класифікації.
5. **Скоротити social pass** — при низькій confidence або вузькому запиті не викликати social agent.
6. **Показувати confidence** — використовувати `confidenceLabel` для візуальної позначки (наприклад, жовтий/зелений бейдж).

### Низький пріоритет
7. **Feedback-кнопка** — “Корисно / Некорисно” для збору метрик.
8. **Експорт** — експорт результату в PDF для подальшої роботи.

## 8. На що звернути увагу при тестуванні

1. **Типи запитів з високим ризиком:**
   - Конкретні дати (наприклад, “26 May 2026”).
   - Конкретні номери статей/annexes.
   - Порівняння MDR vs IVDR (багато тексту, ризик помилок).

2. **Типи запитів, що зазвичай працюють добре:**
   - Загальні питання (“What’s new in MDR 2025?”).
   - Чеклісти (“How to prepare technical documentation?”).
   - Персоналізовані (“How does MDR affect my Class IIb?”).

3. **Сигнали проблем:**
   - body не відповідає на запит.
   - Дублювання однакового тексту для різних запитів.
   - Порожні або нерелевантні links.
   - actionChecklist занадто загальний.

## 9. Підсумок

| Питання | Відповідь |
|---------|-----------|
| **Корисно юзеру?** | Так, як допоміжний інструмент для швидкого орієнтування та планування. |
| **Готово до production?** | З обмеженнями: потрібен disclaimer і перевірка критичних фактів. |
| **Головні ризики** | Відсутність retrieval, потенційні галюцинації, fallback без явної позначки. |
| **Головні напрямки покращення** | Grounding (RAG), фактчекінг дат, явна позначка fallback, збір feedback. |
